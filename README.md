## 🏆 [프로젝트 1] 자동화 도구 비교 구현: GitHub 이슈 관리 워크플로우

### 1. 사용한 도구 이름

- **Make** (구 Integromat)
- **[Relay.app](http://Relay.app)**

### 2. 구현 과정 요약

- **공통 워크플로우:** GitHub에 이슈(Issue)가 생성되면 상세 정보를 가져온 후, '버그' 관련 이슈는 [긴급_버그 트래커] 시트에, 그 외 이슈는 [전체_이슈 백업] 시트에 자동으로 분류/적재하는 파이프라인을 구축함.
- **Make 구현:** `Watch Issues` 트리거와 `Get an Issue` 모듈을 연동하여 배열 데이터를 확보하고, `Router`를 통해 키워드 필터링(버그) 및 `Fallback` 경로(일반 이슈)를 설계함.
- **[Relay.app](http://Relay.app) 구현:** `New Issue` 트리거를 활용하여 선형적인 `Path(If/Else)`를 생성하고, 조건문을 통해 각 시트로 데이터를 배분하는 직관적 워크플로우를 구현함.

### 4. 도구별 장단점 정리

- **Make**
  - **장점:** 정교한 데이터 가공(`map`, `join`, `regex`)이 가능하며, 에러 핸들링을 세밀하게 설계할 수 있음.
  - **단점:** 초기 학습 곡선이 높고 데이터 구조(JSON/Array)에 대한 이해가 필수적임.
- **[Relay.app](http://Relay.app)**
  - **장점:** 별도의 함수 학습 없이 즉시 자동화 구축이 가능하며, 승인(Approval) 프로세스 등 협업 기능이 강력함.
  - **단점:** 복잡한 API 데이터 변환이나 정밀한 파싱 작업에는 다소 제한적임.

### 5. 적합한 상황에 대한 의견

- **Make는 다음과 같은 상황에 적합합니다:**
  - 다수의 외부 API를 복합적으로 연동하고 데이터를 정교하게 가공해야 하는 복잡한 백엔드 자동화.
  - 특정 모듈에서 에러 발생 시 재시도 전략 등을 상세히 설계해야 하는 비즈니스 핵심 프로세스.
- **[Relay.app](http://Relay.app)은 다음과 같은 상황에 적합합니다:**
  - 담당자의 승인이나 확인이 필요한 결재선/협업 위주의 업무 자동화.
  - 코딩 지식이 없는 실무자가 10분 내외로 빠르게 워크플로우를 구축해야 하는 서비스 운영 업무.

### 2. [프로젝트 2] 자유 주제: 폭염 재난 대응 및 긴급 승인 워크플로우

**자동화할 업무:** 매일 아침 기상청 데이터를 조회하여 폭염주의보 발령 기준(일 최고기온 33도 이상 지속) 충족 시, AI가 요약한 경보 문구를 생성하고 관리자의 최종 승인을 거쳐 전파함.

**선정 도구:** **Relay.app** **선정 이유:** 재난 상황 전파 시 시스템의 오작동을 방지하는 'Approval(승인) 노드'가 핵심이며, Relay.app은 이러한 협업 기반 워크플로우 설계에 최적화되어 있기 때문입니다.

### 3. 워크플로우 설계 및 구현 (요구사항 + 보너스 포함)

#### 워크플로우 상세 흐름

1. **Trigger:** `Schedule` (매일 오전 8시 실행)
2. **Action (데이터 조회):** `HTTP - Get Request` (기상청 API 연동, 특정 STN 지역의 오늘 최고기온 조회)
3. **Path (조건 분기):** `If 기온 >= 33도`
  - **Action (보너스 1 - AI 연동):** `OpenAI - Generate Text`
    - "오늘 최고기온이 {{TA}}도입니다. 폭염주의보 수준이니 시민들에게 안내할 안전 문구를 생성해줘."
  - **Action (보너스 2 - 실패 알림 및 재시도):** `Error Handling` 설정
    - AI 생성 실패 시, `Google Sheets - Add Row`를 통해 [긴급_재난_임시저장_시트]에 수동 처리해야 할 Raw 데이터를 저장.
  - **Action (Approval):** `Relay.app - Approval`
    - 관리자(본인)에게 Slack/Email로 발송. [승인] 클릭 시에만 다음 단계 진행.
  - **Action (최종 전파):** `Slack - Send Message` (최종 승인된 문구 전파)

#### 보너스 전략 상세

- **보너스 1 (AI 연동):** 단순 날씨 정보 전달이 아닌, AI가 생성한 '시민 행동 요령'을 포함한 문구를 전송하여 알림의 가치를 높임.
- **보너스 2 (실패/재시도):** Relay.app은 특정 액션이 실패할 경우, `Error Handling` 경로를 지정할 수 있습니다. AI 요약 실패 시 단순히 흐름을 멈추는 것이 아니라, [임시_데이터_적재_시트]에 날짜, 기온, 에러로그를 남겨 데이터 유실을 방지합니다.

### 4. 보고서용 워크플로우 흐름 설명

> **"데이터 정합성 보장을 위한 재난 전파 파이프라인 설계"** 본 프로젝트는 기상청 오픈 API와 자동화 툴을 연동하여, 재난 상황의 1차 감지부터 최종 전파까지의 전 과정을 자동화하였다. 특히, **'인간의 판단(Human-in-the-loop)'** 단계를 강제함으로써 자동화 시스템의 신뢰성을 확보하였다. 실패 시에는 **데이터 적재 방식의 전환(Google Sheets 임시 저장)**을 통해 데이터 유실을 방지하는 방어적 아키텍처를 구현하였다.
